Gherkin BDD

Learn about the key components of Gherkin syntax used in Behavior-Driven Development (BDD).

What is Gherkin?

Gherkin is a business-readable, domain-specific language that helps software teams describe behavior without detailing how that behavior is implemented. It is used in Behavior-Driven Development (BDD) to write acceptance criteria or tests in a way that both technical and non-technical team members can understand.

Key characteristics of Gherkin:

  • Plain Language: Written in a structured but natural language.
  • Line-Oriented: Each line usually starts with a keyword.
  • Structured: Follows a specific syntax to define features, scenarios, and steps.
  • Executable Specifications: Gherkin files can be used as input to testing frameworks (like Cucumber, SpecFlow) to automate tests.

Gherkin Scenario Example

A Scenario is a concrete example that illustrates a business rule. It consists of a list of steps.

Feature: User Login

  Scenario: Successful login with valid credentials
    Given the user is on the login page
    When the user enters "valid_username" as username
    And the user enters "valid_password" as password
    And the user clicks the "Login" button
    Then the user should be redirected to the dashboard
    And the user should see a "Welcome" message

Scenarios are designed to be concise and focused on a single piece of functionality or business rule. They are the core of a Gherkin feature file.

The "Given" Step

Given steps are used to describe the initial context or preconditions of the system before the user (or an external system) starts interacting with it. They put the system into a known state.

  • Sets up the scene for the scenario.
  • Should describe a state, not an action.
  • Multiple Given steps can be used, or they can be joined with And or But.

Example:

Given the user is on the login page
And the user has a valid account

The "When" Step

When steps describe an event or an action performed by the user or an external component. This is the part of the scenario where the interaction with the system occurs.

  • Represents the key action the user performs.
  • Typically, there is one central When step per scenario, though multiple can be used with And or But to describe a sequence of actions.

Example:

When the user enters "test_user" as username
And the user enters "password123" as password
And the user clicks the "Submit" button

The "Then" Step

Then steps are used to describe the expected outcome or result after the action in the When step has been performed. They are used to verify that the system behaves as expected.

  • Asserts the outcome of the When step.
  • Should be an observable result.
  • Multiple Then steps can be used, or they can be joined with And or But to check various outcomes.

Example:

Then the user should be successfully logged in
And the user should be redirected to the account page
And a success message "Login successful!" should be displayed
